# 校园选课记录检索与大数据分析系统

## 📚 项目简介

这是一个用于管理、检索和分析高校学生选课记录的综合系统。系统支持多种数据存储引擎（链表、哈希表、AVL 树），提供数据生成、外部排序、多关键字查询、统计分析等功能。

## ⚠️ 关于测试稳定性的说明

### 数据生成函数可能"失败"的原因

**这是正常现象！** 数据生成器 (`data_generator.c`) 在某些情况下会提前结束生成，原因如下：

1. **课程容量限制机制**：
   - 每门课程每个学期有最大容量限制（默认 180 人）
   - 当某门课的某个学期选满后，继续尝试写入该课程的记录会被拒绝
   - 这是为了模拟真实教务系统的选课逻辑

2. **年份越界保护**：
   - 系统设定的有效年份范围是 2020-2026
   - 随机生成的数据如果超出这个范围会被过滤

3. **高随机性测试模式的特性**：
   - 在生成大量数据（如 10 万条以上）时，由于完全随机，很容易触发上述限制
   - 系统设有"安全阀"机制，连续失败次数过多时会提前终止，防止死循环

**解决方案：多运行几次即可！**
- 基础测试（100~1000 条数据）几乎不会出错
- 压力测试（10 万条以上）如果提前结束，重新运行几次通常能成功
- 这是随机性导致的正常现象，不是 Bug

### 外部排序可能"失败"的原因

外部排序 (`external_sort.c`) 依赖数据生成器产生的 CSV 文件，因此：
- 如果数据生成失败或生成的数据量不足，外部排序会报错
- 临时文件被意外删除也会导致归并失败
- **同样建议多运行几次**

## 🔧 核心功能模块

### 1. 数据生成器 (`data_generator.c`)

**基本逻辑：**
- 生成符合真实教务系统规则的学生选课记录
- 支持两种模式：
  - **真实教务系统模式**（≤50000 条）：模拟学霸/普通/学渣三种学生类型，有必修课补考逻辑、选修课权重偏好
  - **高随机性测试模式**（>50000 条）：完全随机生成，用于压力测试

**数据结构：**
```c
typedef struct {
    char student_id[20];    // 学号
    char name[50];          // 姓名
    char college[50];       // 学院
    char course_id[20];     // 课程号
    char course_name[50];   // 课程名
    float credit;           // 学分
    char semester[20];      // 学期
    char enroll_date[20];   // 选课日期
    int score;              // 成绩（-1 表示缺考）
    int is_elective;        // 是否选修
} CourseRecord;
```

**注释代码 vs 正常代码：**
- **保留注释代码**：代码中包含详细的行内注释，解释每一步的逻辑（如 Box-Muller 正态分布生成算法、轮盘赌选课算法等），适合学习和理解
- **去掉注释代码**：功能完全不变，只是代码更简洁，适合生产环境

---

### 2. 数据存储引擎

系统支持三种存储引擎，通过 `StorageEngine` 枚举切换：

#### (1) 链表引擎 (`linked_list.c`)
- **实现逻辑**：双向链表，支持 O(n) 插入和删除
- **适用场景**：小数据量、顺序遍历
- **注释代码的作用**：注释详细说明了链表节点的摘除和插入过程，帮助理解指针操作

#### (2) 哈希表引擎 (`hash_table.c`)
- **实现逻辑**：拉链法解决冲突，将 (学号，课程号) 组合作为键
- **适用场景**：大数据量的快速查找（O(1) 平均复杂度）
- **注释代码的作用**：解释了哈希值计算过程（融合学号和课程号）、头插法的实现细节

#### (3) AVL 树引擎 (`avl_tree.c`)
- **实现逻辑**：自平衡二叉搜索树，保证 O(log n) 操作
- **适用场景**：需要有序遍历的场景
- **注释代码的作用**：展示了 LL/RR/LR/RL 四种旋转操作的实现

**注释代码 vs 正常代码：**
- **保留注释代码**：可以看到每种数据结构的核心操作（如旋转、哈希计算、链表摘除）的详细步骤
- **去掉注释代码**：功能完全一样，只是失去了学习价值

---

### 3. 外部排序 (`external_sort.c`)

**基本逻辑：**
采用经典的**分块 - 归并**策略：

1. **阶段一：分割（生成初始顺串）**
   - 每次读取 `MEMORY_LIMIT`（默认 1000 条）到内存
   - 使用 `qsort` 在内存中排序
   - 写入临时文件 `temp_r0_XXX.csv`

2. **阶段二：K 路归并**
   - 使用最小堆同时合并 K 个有序文件（默认 K=10）
   - 每轮归并后删除旧的临时文件
   - 重复直到只剩一个文件

**关键数据结构：**
```c
typedef struct {
    char student_id[MAX_STU_ID_LEN];  // 比较键
    char line[1024];                   // 完整 CSV 行
    int file_idx;                      // 来源文件索引
} HeapNode;  // 最小堆节点
```

**注释代码 vs 正常代码：**
- **保留注释代码**：清晰展示最小堆的上浮/下沉操作、双缓冲数组的交换技巧
- **去掉注释代码**：功能不变，但难以理解归并过程

---

### 4. 高级操作 (`advanced_ops.c`)

**功能列表：**
- **多关键字排序**：支持按多个字段（如先按学院、再按学号、最后按成绩）排序
- **数据导出**：将当前数据导出为 CSV
- **过期记录处理**：标记或删除超过一定学期的记录

**多关键字排序的实现逻辑：**
```c
// 通过全局变量传递排序规则（因为 qsort 的比较函数签名固定）
SortRule g_sort_rules[MAX_SORT_RULES];

int MultiKeyCompare(const void *a, const void *b) {
    for (int i = 0; i < g_rule_count; i++) {
        int cmp = CompareByField(a, b, g_sort_rules[i]);
        if (cmp != 0) return g_sort_rules[i].descending ? -cmp : cmp;
    }
    return 0;
}
```

**注释代码 vs 正常代码：**
- **保留注释代码**：解释了为什么需要使用全局变量（qsort 的限制）、浮点数比较的精度处理
- **去掉注释代码**：功能完全相同

## 📝 代码注释说明

本项目的代码注释分为两类：

### 1. 功能性注释（保留时有额外功能）
这类注释通常是**被 `#if 0 ... #endif` 包裹的代码**或**条件编译开关**：
- 启用时可以开启调试输出、性能统计等额外功能
- 禁用时功能精简，运行更快

**示例位置**：
- `data_generator.c` 中的进度打印（某些版本用条件编译控制）
- `external_sort.c` 中的 I/O 统计输出

### 2. 解释性注释（不影响功能）
这类注释占绝大多数，作用是：
- 解释算法原理（如 Box-Muller 变换、最小堆操作）
- 说明业务逻辑（如选课规则、补考逻辑）
- 提示注意事项（如跨平台兼容性）

**去掉这些注释的影响**：
- ✅ 功能完全不变
- ✅ 编译速度略微提升
- ❌ 失去学习价值，难以理解代码意图

## 🐛 常见问题

### Q: 数据生成器提示"生成提前结束"？
**A**: 这是正常现象！原因是课程容量限制或年份越界。解决方法：
1. 减少生成数量（如从 10 万改为 5 万）
2. 多运行几次，随机种子不同可能成功
3. 忽略警告，已生成的数据仍然可用

### Q: 外部排序找不到临时文件？
**A**: 可能是上一轮运行被中断导致。解决方法：
1. 手动删除所有 `temp_r*.csv` 文件
2. 重新运行外部排序

### Q: VSCode 提示"找不到 gcc"？
**A**: 修改 `.vscode/c_cpp_properties.json` 中的 `compilerPath` 为你本地的 gcc 路径

### Q: 编译时出现中文乱码？
**A**: `build.bat` 已设置 `-finput-charset=UTF-8 -fexec-charset=UTF-8`，如果仍有问题：
1. 确保源文件保存为 UTF-8 编码
2. 在 VSCode 右下角切换编码为 UTF-8
